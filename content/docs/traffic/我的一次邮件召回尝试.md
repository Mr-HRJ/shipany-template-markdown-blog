> 这篇文章复盘了我一次**几乎零成本的邮件营销召回实验**：通过 Resend 每日免费额度，累计发送 **190 封邮件**，成功召回 **8 位用户**。重点不在“邮件写得多好”，而在于**用户筛选逻辑、发送节奏控制、落地页面闭环设计**。整套流程我都做成了可复用能力，后续不管是节日运营、版本更新还是二次激活，都可以直接套用。


这次尝试的前提非常简单：

* **成本几乎为 0**
* Resend 每天可免费发送 **100 封邮件**
* 哪怕只召回 1–2 个用户，也是不亏的实验

所以我并没有指望“爆发式转化”，而是想验证一件事：

**邮件召回在小体量产品里，到底有没有实际价值。**

---

# 实验结果先说结论

先给出核心数据，避免后面看得太抽象。

| 指标     | 数据                |
| ------ | ----------------- |
| 累计发送邮件 | 190 封             |
| 召回用户数  | 8 人               |
| 召回方式   | 积分激励 + 登录跳转       |
| 成本     | 0（使用 Resend 免费额度） |

从结果来看，这不是一个“转化率很高”的手段，但它是：

* **稳定的**
* **可复制的**
* **边际成本接近 0 的**

---

# 第一步：只圈“值得被打扰”的用户

邮件召回最容易翻车的一点是：
**给所有人群发。**

我一开始就明确了一个原则：

> 邮件只发给“理论上最可能回来”的那一小撮人。

## 我的用户筛选逻辑

我优先圈了三类用户，**而且是叠加条件**，不是满足其一即可。

### 1. 付费过，但近期没有活跃

这是最优先级的人群：

* 已经认可过产品价值
* 只是阶段性流失
* 回来的心理成本最低

### 2. 使用过产品，但目前不活跃

这类用户至少完成过：

* 注册
* 实际使用

说明不是“随手点进来就走”的路人。

### 3. IP 来自高价值地区

我额外加了一层过滤：

* 只选高价值地区 IP
* 直接排除低转化区域

最终筛选出来的用户数量并不多，但**质量非常集中**。

这些用户我统一存进数据库，并且设置为：

* 只有这批用户
* 才能领取后续赠送的积分

---

# 第二步：自己做一个“批量发邮件”的页面

直接用 Resend API 发邮件是没问题的，但对运营来说非常不顺手。

所以我干脆：

**让 AI 给我写了一个批量发送邮件的页面。**

## 这个页面做了什么

逻辑非常简单：

* 从数据库读取已圈定用户
* 每 **30 秒** 调用一次 Resend API
* 一次只发 **1 封邮件**

这样做有几个好处：

* 避免短时间大量请求
* 更像“人工发送”，降低风控风险
* 可以随时中断、调整内容

---

## 邮件内容本身怎么写

邮件内容不是我自己硬写的，而是：

* 让 AI 生成 **HTML 邮件**
* 重点突出一件事：**赠送积分**
* 文案目标只有一个：**把人拉回来登录**

没有复杂介绍，没有长篇营销。

---

# 第三步：专门为召回做一个落地页面

如果邮件里只是一个普通登录链接，转化会非常低。

所以我让 AI **单独写了一个落地页面**，专门服务这次召回。

## 落地页面的流程设计

整个流程非常短：

1. 用户点击邮件中的链接
2. 进入“领取积分”页面
3. 登录后自动领取积分
4. 自动跳转到核心功能页面

这里有两个关键点：

* **积分是立即到账的**
* **用户会被直接带回产品使用场景**

不是领完就结束，而是顺手继续用。


---

# 为什么我要把这套流程做成“能力”

这次实验结束后，我并没有把代码丢一边。

原因很简单：

**这套东西以后一定还能用。**

## 可复用的场景非常多

比如：

* 节日运营
* 新功能上线
* 定价调整
* 老用户专属福利

每次只需要改三样东西：

* 用户筛选规则
* 邮件文案
* 奖励内容

流程本身完全不用重写。

---

# 我对这次邮件召回的判断

结合数据和实际体验，我的结论是：

* 邮件召回不适合“拉新”
* 但非常适合**激活已经被你筛选过的用户**
* 成本低到几乎可以忽略

它更像是一种：

> **“你还在吗？我这边有点新东西。”**

而不是促销轰炸。

---

## 总结

如果你已经有：

* 用户数据库
* 基础的邮件发送能力
* 能给用户一点点实际激励

那邮件召回非常值得你试一次。

哪怕结果只是：

* 发 100 多封
* 回来几个人

也完全不亏。

很多用户不是不想用你，
而是你**从来没有给过他们一个回来用的理由。**
